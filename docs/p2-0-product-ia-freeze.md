# X-P2-0 P2 产品信息架构与任务池冻结报告

## 目的

本文对 `docs/p2-product-direction-ia.md` 和 `docs/wiki.html` 中 P2 任务池做总控复核，确认 PLS P2 产品方向、一级模块、做 / 不做范围和 D/A/M/V 任务边界是否可以冻结。

## 结论

结论：通过，P2 产品信息架构与任务池冻结。

`docs/p2-product-direction-ia.md` 可作为 P2 产品口径来源；`docs/wiki.html` 中 X-P2-0 至 V-P2-11 可作为后续派发任务池。后续实现必须遵循店铺 / 账号优先、解释型匹配优先、真实数据可追溯和不自动决策的约束。

## 冻结口径

P2 产品定位冻结为：

> 面向零售品牌多角色用户的「渠道 / 店铺 / 账号 × 商品 × 人群 × 行动反馈」业务智能 BI 系统，具备新品人群预测与解释型人货匹配能力，并通过数据接入和经营反馈逐步沉淀决策记忆。

P2 一级模块冻结为：

| 模块 | 冻结口径 |
|---|---|
| 数据管理 | 数据源、导入批次、版本、字段映射、质量和审计底座 |
| 渠道人群 | 以店铺 / 账号 / 门店为第一实体查看人群画像 |
| 商品人群 | 覆盖在售、历史已售品类、SPU、SKU 的真实消费人群 |
| 人货匹配 | 核心决策模块，输出匹配分、解释、置信度和风险 |
| 新品预测 | 基于商品主数据预测人群画像，并接入匹配链路 |
| 经营飞轮 | 记录决策、行动、反馈和策略调整，先做复盘闭环 |

P2 主路径冻结为：

```text
数据管理 -> 渠道人群 / 商品人群 -> 人货匹配 -> 新品预测 -> 经营飞轮
```

## 做 / 不做范围

P2 要做：

- 把数据管理作为产品底座，支持真实数据导入、版本、质量和审计。
- 把线上店铺 / 账号、线下门店作为渠道人群第一分析实体。
- 把人货匹配升级为核心决策入口，支持按商品找渠道和按渠道找商品。
- 新品预测以商品主数据为主要输入，先冻结输入 / 输出 contract 和可解释 baseline。
- 经营飞轮先记录决策、行动、反馈和复盘状态。

P2 初期不做：

- 不做通用 BI 平台。
- 不做自动决策、自动投放、自动上架或自动策略执行。
- 不编造用户尚未提供的新品商品主数据字段、枚举、默认值或业务 ID。
- 不把 `legacyFitScore` 当成正式产品算法结论。
- 不用 iframe、外链、截图、静态整页 HTML 或前端内嵌 `data.js` 承担产品主流程。

## 任务边界

| 任务 | 结论 |
|---|---|
| `A-P2-1` | 数据管理底座与导入审计 API，可先做读取型 API 和 contract，再评估写入型 HTTP import / rollback |
| `D-P2-2` | 商品主数据与渠道实体 schema 草案，是后续渠道、商品、新品预测的结构前置 |
| `A-P2-3` / `V-P2-4` | 店铺 / 账号优先的渠道人群 API 与工作台，需避免平台渠道作为唯一主轴 |
| `M-P2-5` / `V-P2-6` | 解释型匹配 contract 与核心工作台，正式 fit formula 未冻结前保留替换点 |
| `D-P2-7` / `M-P2-8` / `A-P2-9` | 新品主数据预测链路，用户提供字段后再落具体 schema 和 fixture |
| `A-P2-10` / `V-P2-11` | 经营飞轮最小闭环，只做记录、反馈和复盘，不做自动执行 |

## 风险与待拍板

- 正式 fit formula 尚未冻结，M 域只能先维护解释型 contract 和 baseline adapter。
- 新品商品主数据字段尚未提供，D/M/A 只能先冻结字段组、输入质量和 contract。
- 抖音 BI `douyin_*` 数据是否投影到通用 `channel_profile` 仍需 A-P2-3 单独给方案。
- A-P2-1 第一阶段建议先做读取型数据管理 API；HTTP import、rollback 写路径可作为同卡第二阶段或拆后续卡。

## 验收

- P2 信息架构冻结结论明确。
- `docs/wiki.html` 中 P2 任务卡可被 D/A/M/V 复制执行。
- 后续实现约束已明确：店铺 / 账号优先、解释型匹配、数据可追溯、P2 初期不自动决策。
