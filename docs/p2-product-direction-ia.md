# PLS P2 产品方向与信息架构草案

## 目的

本文冻结 PLS 从 P1-F 抖音 BI 产品化验收后进入 P2 的产品方向草案，用于后续拆任务卡、设计数据契约和规划界面主路径。

PLS 的定位调整为：

> 面向零售品牌多角色用户的「渠道 / 店铺 / 账号 × 商品 × 人群 × 行动反馈」业务智能 BI 系统，具备新品人群预测与解释型人货匹配能力，并通过数据接入和经营反馈逐步沉淀决策记忆。

PLS 不做通用 BI，也不只做单点模型工具。P2 的产品主线是：以数据管理为底座，以店铺 / 账号优先的人群画像为资产，以解释型人货匹配为核心决策入口，以新品预测和经营飞轮承接从判断到反馈的闭环。

## 使用方式

- 总控拆 P2 任务前，先用本文确认一级模块、核心用户故事和不做范围。
- D / A / M / V 域设计 schema、API、模型 contract 和页面时，以本文的对象关系和模块边界为产品口径。
- `docs/wiki.html` 仍是任务派发真源；本文不是任务状态来源，确认后再拆为 wiki 任务卡。
- 若用户后续提供新品商品主数据字段、真实渠道数据接口或行动反馈数据，先更新数据契约，再进入实现任务。

## 产品原则

- 店铺 / 账号优先：线上渠道以单店铺、单账号为第一分析实体，平台渠道只作为分组维度；线下渠道以省份、城市、商圈、门店逐层展开。
- 解释型优先：匹配和预测先输出分数、理由、标签差异、置信度和风险提示，不直接自动决策；沉淀足够决策记忆后再考虑半自动或自动策略。
- 多角色共用：品牌老板、经营负责人、渠道运营、商品企划、投放团队都是目标用户；每个核心模块都应同时支持决策视图和分析视图。
- 数据可追溯：所有画像、匹配、预测和反馈结果必须保留来源、批次、时间窗口、样本量、置信度或质量标记。
- 真实数据接入：真实业务数据通过导入数据包或连接数据接口进入系统；导入、版本、校验和审计能力是 P2 底座，不是附属工具。

## 一级模块

### 1. 数据管理

定位：PLS 的数据接入、版本管理和质量控制入口。

核心问题：

- 当前系统接入了哪些数据源？
- 哪些批次可用、失败、过期或质量不足？
- 字段如何映射到 PLS 的商品、渠道、画像、行动和反馈对象？
- 最新数据和历史版本如何并存、回滚和审计？

P2 范围：

- 数据源列表：商品主数据、历史销售数据、渠道 / 店铺 / 账号画像、行动反馈、外部 BI 数据源。
- 导入记录：批次、版本、状态、导入时间、操作者、来源系统。
- 字段映射：源字段、目标字段、映射规则、未映射原因。
- 数据质量：缺失率、覆盖率、样本量、异常值、低置信度标签。
- 版本管理：latest projection、历史版本查询、回滚候选、导入审计。

### 2. 渠道人群

定位：查看渠道、店铺、账号、城市商圈或门店的人群画像。

核心问题：

- 这个店铺 / 账号 / 门店的人群是谁？
- 人群结构是否稳定，最近是否变化？
- 与平台平均、品牌平均或标杆账号相比差异在哪里？
- 哪些标签置信度高，哪些只是弱信号？

P2 范围：

- 线上：天猫、京东、抖音、视频号等平台下的店铺、账号、直播间或内容账号。
- 线下：省份、城市、商圈、门店。
- 支持画像标签、销售表现、内容表现、转化表现和 benchmark tags。
- 支持决策视图和分析视图。

### 3. 商品人群

定位：查看在售或历史已售品类、系列、SPU、SKU 的真实消费人群。

核心问题：

- 这个品类或单品实际卖给了谁？
- 不同渠道下同一商品的人群是否不同？
- 历史爆品和滞销品的人群差异是什么？
- 哪些商品可作为新品预测的相似样本？

P2 范围：

- 品类、系列、SPU、SKU 多粒度画像。
- 商品主数据、销售表现、真实画像标签、质量标记。
- 支持跨渠道对比和单渠道钻取。
- 支持相似商品候选，用于新品预测和解释。

### 4. 人货匹配

定位：PLS 的核心决策模块，解释商品与店铺 / 账号 / 门店的适配关系。

核心问题：

- 这个商品最适合哪些店铺 / 账号 / 门店？
- 这个店铺 / 账号最适合卖哪些商品？
- 匹配高是因为哪些标签相似？
- 匹配低是因为哪些标签冲突或数据不足？

P2 范围：

- 匹配分：保留 `legacyFitScore` 的诊断参考属性，后续替换为正式 fit formula。
- 解释层：相似标签、冲突标签、缺失标签、低置信度标签、推荐理由、风险提示。
- 视角切换：按商品找渠道，按渠道找商品。
- 输出约束：不做自动决策，只给解释型建议和优先级。

### 5. 新品预测

定位：基于新品商品主数据预测目标人群，并给出渠道 / 店铺 / 账号适配建议。

核心问题：

- 新品未上市时，预计会吸引什么人群？
- 它与哪些历史商品相似？
- 适合先在哪些店铺 / 账号 / 门店试销或投放？
- 预测置信度和不可用边界是什么？

P2 范围：

- 输入：商品主数据。具体字段由用户后续提供，初步预留类目、价格带、季节、卖点、材质、风格、目标场景、图文资产、相似 SKU 等字段组。
- 输出：预测画像、相似历史商品、适配渠道建议、置信度、解释依据、风险提示。
- 接入人货匹配：新品预测结果可作为商品画像输入，进入匹配模块。
- 不在 P2 初期追求复杂自动模型，先冻结输入 / 输出 contract 和可解释 baseline。

### 6. 经营飞轮

定位：把决策、行动、反馈和策略调整记录为闭环，长期沉淀决策记忆。

核心问题：

- 当时基于什么判断做了什么行动？
- 行动后销售、转化、投放、人群结构有什么变化？
- 原始判断被验证、偏离还是失败？
- 下一轮策略应该如何调整？

P2 范围：

- 决策记录：目标商品、目标店铺 / 账号、推荐依据、负责人、时间。
- 行动记录：上架、铺货、投放、内容、直播、活动、价格策略等。
- 反馈导入：销量、GMV、转化、ROI、退货、复购、人群偏差。
- 复盘状态：待执行、执行中、待复盘、已验证、需调整。
- P2 初期只做最小闭环，不做自动策略。

## 用户角色

| 角色 | 主要问题 | 重点视图 |
|---|---|---|
| 品牌老板 | 哪些渠道和商品值得优先投入？ | 决策视图、趋势、风险、优先级 |
| 经营负责人 | 商品、渠道和行动是否形成闭环？ | 匹配结果、行动反馈、复盘状态 |
| 渠道运营 | 某店铺 / 账号适合什么货，如何调整？ | 渠道人群、按渠道找商品、执行建议 |
| 商品企划 | 新品面向谁，和历史商品像不像？ | 商品人群、新品预测、相似 SKU |
| 投放团队 | 哪些人群标签和渠道触点值得投？ | 标签解释、账号画像、反馈 ROI |

## 核心用户故事

- 作为经营负责人，我希望看到某新品适合先投放到哪些店铺 / 账号，并理解推荐原因和风险。
- 作为渠道运营，我希望查看某个账号的人群画像，并找到最适合该账号承接的商品。
- 作为商品企划，我希望查看历史已售商品的人群差异，用于定义新品目标人群。
- 作为投放团队，我希望知道商品与账号匹配的相似标签和冲突标签，避免只看到一个黑盒分数。
- 作为总控或数据负责人，我希望查看每个数据批次的来源、版本、质量和字段映射，避免不可追溯的分析结论。
- 作为经营负责人，我希望把一次匹配建议转成行动记录，并在导入反馈后看到判断是否成立。

## 数据对象关系

```text
DataSource
  -> ImportBatch
  -> DataVersion
  -> FieldMapping
  -> DataQualityReport

ProductMaster
  -> ProductProfile
  -> PredictedProductProfile

ChannelEntity
  -> ChannelProfile

ProductProfile + ChannelProfile
  -> ProductChannelFit
  -> FitExplanation

DecisionRecord
  -> ActionRecord
  -> FeedbackRecord
  -> StrategyAdjustment
```

建议对象说明：

| 对象 | 说明 |
|---|---|
| `DataSource` | 一个外部系统、文件包或人工维护数据源 |
| `ImportBatch` | 一次导入动作，记录状态、来源、时间和操作者 |
| `DataVersion` | 可查询的业务数据版本，支持 latest 和历史版本并存 |
| `FieldMapping` | 源字段到 PLS 目标字段的映射关系 |
| `DataQualityReport` | 批次级或对象级质量报告 |
| `ProductMaster` | 商品主数据，作为商品画像和新品预测输入 |
| `ProductProfile` | 在售或历史已售商品的人群画像 |
| `PredictedProductProfile` | 新品预测人群画像 |
| `ChannelEntity` | 店铺、账号、直播间、城市商圈、门店等可承接销售的实体 |
| `ChannelProfile` | 渠道实体的人群画像 |
| `ProductChannelFit` | 商品与渠道实体的匹配结果 |
| `FitExplanation` | 匹配分的解释层，含相似、冲突、缺失和低置信度信号 |
| `DecisionRecord` | 基于系统判断形成的经营决策 |
| `ActionRecord` | 决策后的执行动作 |
| `FeedbackRecord` | 真实销售、投放、转化或人群反馈 |
| `StrategyAdjustment` | 复盘后的策略调整 |

## P2 任务拆分建议

1. `X-P2-0`：P2 产品信息架构与任务池冻结
   - 产出：确认本文口径，拆出 D / A / M / V 可执行任务卡。
   - 验收：wiki 新增 P2 卡组，明确做 / 不做范围。

2. `A-P2-1`：数据管理底座和导入审计 API
   - 产出：数据源、导入批次、版本查询、导入审计、回滚候选 API。
   - 验收：支持现有抖音 BI CLI 导入结果查询，并预留 HTTP import endpoint。

3. `D-P2-2`：商品主数据与渠道实体 schema 草案
   - 产出：`ProductMaster`、`ChannelEntity`、字段映射模板和质量报告模板。
   - 验收：可覆盖线上店铺 / 账号和线下省市商圈门店。

4. `A/V-P2-3`：店铺 / 账号优先的渠道人群工作台
   - 产出：以店铺 / 账号为第一实体的查询 API 和前端视图。
   - 验收：不再把平台渠道作为唯一主轴，支持 account / shop drilldown。

5. `M-P2-4`：解释型人货匹配 contract
   - 产出：正式 fit formula 待冻结前的解释对象、置信度和风险输出 contract。
   - 验收：覆盖 matched、mismatch、low confidence、unmapped、insufficient sample。

6. `V-P2-5`：人货匹配核心工作台
   - 产出：按商品找渠道、按渠道找商品、解释面板和状态页。
   - 验收：真实 API 下通过 desktop / mobile smoke，覆盖 empty / error / low confidence / unmapped。

7. `D/M/A-P2-6`：新品主数据预测入口
   - 产出：新品输入 contract、预测输出 contract、baseline adapter 和 API。
   - 验收：用户提供商品主数据字段后，可跑通一条新品预测到匹配建议链路。

8. `A/V-P2-7`：经营飞轮最小闭环
   - 产出：决策记录、行动记录、反馈导入、复盘状态。
   - 验收：一次匹配建议可转行动，导入反馈后进入复盘视图。

## 示例

新品预测到匹配的最小流程：

```text
1. 数据管理导入商品主数据和账号画像。
2. 新品预测根据 ProductMaster 生成 PredictedProductProfile。
3. 人货匹配把 PredictedProductProfile 与 ChannelProfile 计算为 ProductChannelFit。
4. 用户查看 FitExplanation，选择目标店铺 / 账号。
5. 经营飞轮生成 DecisionRecord 和 ActionRecord。
6. 后续导入销售或投放反馈，形成 FeedbackRecord。
7. 系统标记判断成立、偏离或需调整。
```

按账号找商品的最小流程：

```text
1. 渠道运营打开某抖音账号或天猫店铺。
2. 系统展示该账号 / 店铺的 ChannelProfile、表现指标和 benchmark 差异。
3. 用户切到“适合商品”视角。
4. 系统按 ProductChannelFit 排序推荐商品，并展示相似标签、冲突标签和置信度。
5. 用户将候选商品加入行动计划。
```

## 注意事项

- 本文是产品方向草案，不替代 `docs/wiki.html` 的任务派发状态。
- P2 初期不做自动决策，不把模型输出直接变成执行命令。
- 新品商品主数据字段尚未冻结，不能提前编造字段枚举、默认值或业务 ID。
- 真实数据接入可以通过数据包或接口，但必须保留来源、版本、批次、质量和字段映射。
- 当前抖音 BI 数据仍保留在独立 `douyin_*` 表和 `/api/v0/bi/douyin/*` 主流程；是否同步进通用 `channel_profile` 需要在 P2 任务中单独拍板。
- `legacyFitScore` 继续作为诊断参考，不作为正式产品算法结论；正式 fit formula 待用户或 M 域后续冻结。
- 飞轮闭环在 P2 先做记录与复盘，不做自动策略优化。
