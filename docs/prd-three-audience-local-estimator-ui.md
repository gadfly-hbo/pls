# 三大人群本地估算界面 PRD

## 目的

在渠道对象详情页的“编辑”Tab 后新增“三大人群”Tab。用户可在浏览器中选择本地 CSV、Markdown 或 XLSX 人群画像文件，确认识别出的原生人群标签与占比后，选择渠道算法并得到森马三大人群占比估算结果。

## 使用方式

1. 在渠道对象详情页打开“三大人群”。
2. 选择一个 CSV、MD 或 XLSX 文件，或选择文件夹后从可识别文件清单中选定一个文件。
3. 用户选择渠道；页面读取 CSV/XLSX 的首张表或 Markdown 的首个表格，并识别“人群标签 + 占比”两列。
4. 用户确认或更正列映射后，页面只保留所选渠道算法支持的原生人群标签及已冻结别名，并展示已忽略的无关行数。
5. 页面展示 A「质感流行派」、B「都市体面家」、C「百搭优选客」占比，以及 coverage、uncovered、算法版本、质量标记和未映射标签。

## 输入契约

- 每次计算只使用一个文件；文件夹只用于列出候选文件，不自动合并多个文件。
- CSV/XLSX 只读取首张表；Markdown 只读取第一个 GFM 表格。
- 只接受两列业务含义：人群标签和占比。可自动候选的列名仅限：
  - 标签列：`人群标签`、`标签`、`人群`、`label`、`segment`
  - 占比列：`占比`、`比例`、`渗透率`、`share`、`ratio`
- 自动识别结果必须呈现给用户确认；未识别、重复、空值、非数值、负数或大于 100% 的行必须阻止计算并指出行号或原因。
- 占比可填写 `0-1` 小数或带 `%` 的百分数；解析后统一转换为 `0-1`。
- 筛选后的输入总和最多可为约 `100.1%` 的来源四舍五入误差；容差内先归一化，超过容差才阻止计算。前端必须复用模型层容差，不得维护独立阈值。
- 先按所选渠道的原生人群体系筛除无关行，再做重复、数值和总和校验。例如抖音只保留八大人群及已冻结别名；兴趣、地域等其他标签维度不得造成重复标签或总和超额错误。

## 算法与展示

- 渠道选项固定为抖音、天猫、京东、线下、唯品会、视频号、拼多多，并由页面确定对应原生体系。
- 必须复用 `apps/model/src/three-audience-share.ts` 的 `estimateSemirThreeAudienceShares`，不得在 UI 中复制七渠道映射矩阵、京东校准矩阵或算法公式。
- 必须复用模型层的渠道原生标签识别入口筛选文件行；不得在 UI 中复制标签集合或别名规则。
- 默认不输入专家先验；允许用户在当前页面输入 A/B/C 先验并重算。先验不写入 URL、对象、API、数据库或本地存储。
- 结果不写入渠道对象，不作为当前“人群画像”Tab 的数据来源。
- 页面要清楚标明“本次计算仅在当前浏览器会话保留”。

## 非目标

1. 不新增 Server API、数据库 schema、导入任务、对象版本或文件上传。
2. 不定义渠道、店铺、账号画像的持久化结构。
3. 不支持多文件合并、历史结果、批量估算或自动选择渠道。
4. 不从年龄、偏好、兴趣或其他非原生人群份额推断标签。
5. 不支持 PDF、图片、HTML、JSON 或任意本地绝对路径读取。

## 实现与验证边界

- 前端可在 `apps/web/package.json` 声明项目已有的 `xlsx@^0.18.5`，仅用于浏览器端 XLSX 解析；CSV 和 Markdown 不依赖该库。
- 必须保持窄屏无横向溢出，并为长文件名、错误列表和未映射标签提供换行或横向滚动保护。
- 至少新增 Playwright 覆盖：Tab 可达、CSV 解析后的列确认、渠道选择、结果/coverage/质量标记展示、非法输入阻止计算、无后端请求。
- 必须运行 `npm run build`（`apps/web`）及定向 Playwright 测试；模型算法以既有 `npm run three-audience-share-contract-test` 作为复用回归证据。

## 注意事项

- 本 UI 的输入格式只是本地临时估算格式，不等同于后续要冻结的渠道、店铺或账号画像结构。
- 浏览器不允许直接访问任意本地路径；只能由用户通过文件或文件夹选择器授权读取。
