# P0 开工任务书

> 配套 `Orchestration.md` 与 `AGENTS.md` 使用。P0 的目标不是做完整 SaaS，而是跑通“新品画像预测 + 渠道人货匹配”的最小可信闭环。

---

## 一、P0 总目标

在 4-6 周内完成一个可演示、可回测、可继续扩展的 MVP：

1. 输入一个新品 SKU 的基础信息和图文资料。
2. 系统预测该商品的 Top 3 人群画像包。
3. 导入若干渠道画像。
4. 系统计算商品与渠道的匹配度。
5. 输出分货、投流、熔断或换货盘建议。

---

## 二、P0 验收标准

P0 完成时必须具备：

- 一套统一画像标签体系 v0.1。
- 一份历史 SKU 训练宽表规范。
- 一批脱敏 demo 数据。
- 一个新品画像预测接口或可运行脚本。
- 一个商品 × 渠道匹配接口或可运行脚本。
- 一个前端原型或页面，能展示预测画像和匹配热力图。
- 一份回测报告，至少说明 Top 标签命中率、主要误差来源、不可用边界。

---

## 三、任务拆解

### X-P0-0：统一画像标签体系与红线

归属：X 总控

目标：

- 冻结 P0 使用的画像标签坐标系。
- 明确 DMP 标签到内部标签的映射原则。
- 明确原始数据、聚合数据、衍生产物的使用边界。

产出：

- `docs/profile-taxonomy-v0.md`
- `docs/data-safety-policy.md`
- D/M/A/V 的接口输入输出约束。

验收：

- D/M/A/V 任务均能引用同一套画像标签，不各自造口径。

### D-P0-1：历史 SKU 宽表与 DMP 映射

归属：D 数据画像

目标：

- 定义训练宽表字段。
- 设计 DMP 聚合画像导入格式。
- 设计商品基础属性字典。

产出：

- `docs/data-spec.md`
- demo 宽表字段说明。
- 数据质量检查规则。

验收：

- 任一历史 SKU 可表示为 `SKU_ID + 商品DNA + 真实画像标签分布 + 销售表现`。

### M-P0-2：商品画像预测与匹配算法

归属：M 模型预测

目标：

- 设计商品 DNA 到消费者画像的预测方法。
- 输出 Top 3 人群包、置信度、关键驱动标签。
- 设计商品画像与渠道画像匹配度算法。

产出：

- `docs/model-plan.md`
- 预测输出 JSON schema。
- 匹配度计算说明。

验收：

- 可对 demo 新品输出预测画像。
- 可对 demo 商品与 demo 渠道输出匹配分数和原因。

### A-P0-3：API 与 pipeline 契约

归属：A 应用后端

目标：

- 设计 P0 核心 API。
- 设计任务状态机和结果存储。
- 设计审计日志与数据来源追踪。

产出：

- `docs/api-contract.md`
- `docs/pipeline-design.md`

验收：

- V 前端能基于契约画出页面流程。
- M 模型输出能被 A 封装成稳定接口。

### V-P0-4：新品画像与渠道匹配工作台

归属：V 前端决策

目标：

- 设计新品画像预测工作台。
- 设计渠道匹配热力图。
- 设计运营建议输出形态。

产出：

- `docs/ui-flow.md`
- `docs/decision-output.md`

验收：

- 用户能从“录入新品”走到“查看渠道推荐和投流建议”。

### X-P0-5：端到端验收

归属：X 总控

目标：

- 验收 P0 闭环。
- 检查数据红线。
- 归档 wiki changelog 和 notes。

产出：

- P0 验收报告。
- P1 任务拆解。

---

## 四、P0 不做

- 不做全品类，只先以服装为核心样例。
- 不做多租户权限体系，只做最小工作区隔离设计。
- 不接真实平台 API，先定义导入格式和 mock/demo 数据。
- 不自动训练大模型，先做可解释的 tabular baseline 和回测框架。
- 不做投放平台自动执行，只输出可执行建议。
